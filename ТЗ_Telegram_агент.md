# ТЗ — Telegram-агент будівник таблиць

> Версія 1.0 | Фінансова система малого бізнесу | @matsukoleksandr

---

## Контекст

Це не просто бот який приймає JSON і будує таблицю.  
Це **агент** який веде діалог, збирає потрібну інформацію, перевіряє себе і тільки тоді будує.

Агент використовується в кількох уроках курсу:
- **2.2** — таблиця Cashflow
- **3.3** — таблиця P&L
- **5.2** — таблиця Балансу
- **5.4** — Дашборд
- **4.1** — повна архітектура збору даних (Cashflow + P&L разом)

---

## Стек

- Node.js + Express (вже є)
- Telegram Bot API через webhook (вже є)
- Claude API (`claude-sonnet-4-20250514`) — мозок агента
- Google Apps Script Web App — виконавець побудови таблиць
- Змінні середовища: `ANTHROPIC_API_KEY`, `APPS_SCRIPT_URL`

---

## Загальна архітектура агента

```
Користувач
    ↓ текст / code-блок
Telegram-агент
    ↓ аналіз + діалог (Claude API)
    ↓ збір даних через питання
    ↓ формування payload
    ↓ самоперевірка перед відправкою
Apps Script Web App
    ↓ побудова таблиці
    ↓ JSON з посиланнями
Telegram-агент
    ↓ форматований результат
Користувач
```

## 1. Стан агента (в пам'яті процесу per chatId)

```js
{
  stage: 'idle' | 'collecting' | 'confirming' | 'building' | 'editing',
  report_type: null | 'cashflow' | 'pl' | 'balance' | 'dashboard',
  raw_input: '',          // весь текст від користувача накопичується
  extracted: {},          // те що агент витягнув з тексту
  clarifications: [],     // відповіді на питання
  payload: null,          // фінальний payload для Apps Script
  spreadsheet_id: null,   // id побудованого файлу (для правок)
  history: []             // conversation history для Claude
}
```

При `/clear` — повний скид стану.  
Стан живе в пам'яті процесу (без БД) — це прийнятно для поточного етапу.

## 2. Системний промпт агента

Передається як `system` в кожен виклик Claude API.

```
Ти — агент який допомагає побудувати фінансову таблицю Google Sheets для малого бізнесу.

Твоя задача: зібрати достатньо інформації щоб побудувати правильну таблицю, і не більше.
Не питай зайвого. Питай тільки те що реально впливає на структуру таблиці.

Принципи роботи:
1. Спочатку витягни з опису користувача все що вже відомо
2. Визнач що ще потрібно — і питай по одному або по 2-3 пов'язані питання разом
3. Після кожної відповіді оновлюй своє розуміння
4. Коли даних достатньо — сформуй payload і попроси підтвердження
5. Відповідай українською, коротко і по справі

Контекст курсу:
- Учасник — власник малого бізнесу, не технічна людина
- Таблиці будуються на Google Sheets через Apps Script
- Дані вносять різні члени команди з різним рівнем доступу
- Головна складність: визначити хто, як і звідки вносить дані

Формат відповідей:
- Ніяких markdown в Telegram (зірочки, решітки і т.д.)
- Короткі речення
- Якщо кілька питань — нумерований список
```

## 3. Потік роботи агента

### 3.1 Перше повідомлення — аналіз і витяг даних

Користувач надсилає перше повідомлення (текст або code-блок `tz`).

Агент викликає Claude з таким промптом:

```
Користувач описує таблицю яку треба побудувати:

---
{user_message}
---

Твоя задача:
1. Визнач тип таблиці (cashflow / pl / balance / dashboard / невідомо)
2. Витягни все що вже відомо: назва бізнесу, статті доходів і витрат, відповідальні, кількість операцій, доступ до Sheets
3. Визнач що критично відсутнє для побудови таблиці
4. Задай питання тільки про відсутнє — не більше 3 за раз

Відповідай у форматі JSON:
{
  "understood": {
    "report_type": "cashflow|pl|balance|dashboard|unknown",
    "business_name": "...",
    "articles": { "inflows": [...], "outflows": [...] },
    "responsible": { "стаття": { "name": "...", "access": true/false/null } },
    "ops_per_month": { "стаття": число },
    "options": {}
  },
  "missing_critical": ["що відсутнє і критично для побудови"],
  "missing_optional": ["що відсутнє але можна дізнатись пізніше"],
  "message_to_user": "Текст який показати користувачу з питаннями (без markdown)"
}
```

Агент:
- Зберігає `understood` в `state.extracted`
- Показує користувачу `message_to_user`
- Переходить в стадію `collecting`

---

### 3.2 Збір даних — питання про рух грошей

Це ключовий момент якого немає в більшості ботів.

Для кожної статті витрат де `access: false` або `access: null` агент питає **два різних питання**:

**Питання 1 — хто платить (рух грошей):**
```
"[Стаття] — хто фактично платить?
1. [Ім'я] отримує підзвітні кошти і сам розраховується
2. [Ім'я] подає заявку, бухгалтер проводить оплату"
```

Якщо відповідь 2 (бухгалтер платить) → ця людина з введення даних **випадає**,  
бухгалтер вносить сам → питання про доступ до Sheets для цієї людини не задається.

Якщо відповідь 1 (підзвітні) → задається питання 2.

**Питання 2 — як вносить дані (тільки якщо підзвітні):**
```
"[Ім'я] розраховується сам. Як йому зручніше фіксувати витрати?
1. Google Form (просто заповнює форму на телефоні)
2. Окремий простий аркуш у таблиці"
```

---

### 3.3 Визначення архітектури

Після збору даних агент автоматично визначає архітектуру для кожної групи статей:

| Умова | Архітектура |
|-------|-------------|
| ops ≤ 50, всі мають доступ | **A** — прямий ввід |
| ops 50–200 або є хтось без доступу з підзвітними | **B** — окремі вкладки |
| ops > 200 або 3+ людини без доступу | **C** — Google Form + лог |

Архітектури комбінуються: надходження можуть бути A, витрати — B і C одночасно.

Агент **не питає** про архітектуру — він визначає її сам і повідомляє:

```
Проаналізував. Ось що виходить:

Надходження (Марина, 16 оп/міс, є доступ) → пряме введення
Витрати — Зарплати (Наталія, є доступ) → пряме введення
Витрати — Підрядники (Дмитро, підзвітні, 15 оп/міс) → окремий аркуш
Витрати — Реклама (Ірина, підзвітні, 20 оп/міс) → Google Form

Ще кілька питань:
1. Додати аркуш Платіжного календаря?
2. Є кілька банківських рахунків?
3. Потрібен облік по контрагентах (хто конкретно платив)?
```

---

### 3.4 Самоперевірка перед відправкою

Перед тим як сформувати фінальний payload — агент викликає Claude для перевірки:

```
Ось що я зібрав для побудови таблиці Cashflow:

{JSON з зібраними даними}

Перевір:
1. Чи є у кожної статті відповідальний?
2. Чи немає статей з access: null (не визначено)?
3. Чи визначена архітектура для кожної групи статей?
4. Чи є назва бізнесу для іменування файлу?
5. Чи є email для прав доступу до файлу?

Якщо щось відсутнє — поверни список питань.
Якщо все є — поверни: {"ready": true}
```

Якщо `ready: true` — переходить до підтвердження.  
Якщо є питання — задає їх користувачу і залишається в `collecting`.

---

### 3.5 Підтвердження перед побудовою

Агент показує summary і питає підтвердження:

```
Готовий будувати. Ось що вийде:

Файл: Cashflow_Агенція Ткаченко_2026
Аркуші: Cashflow, Надходження, Витрати, Витрати — Дмитро, Google Form для Ірини, Платіжний календар, Довідники, Налаштування

Статті надходжень: Оплата SMM, Оплата реклама, Передоплати
Статті витрат: Зарплати, ЄСВ, Підрядники, Реклама клієнтів, Оренда, SaaS

Будуємо? (так / ні / змінити щось)
```

Якщо "ні" або "змінити" — повертається до `collecting`.  
Якщо "так" — переходить до `building`.

### 3.6 Побудова — виклик Apps Script

```js
const response = await fetch(process.env.APPS_SCRIPT_URL, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify(payload)
});
const result = await response.json();
```

Під час побудови — надіслати повідомлення: `"Будую таблицю, це займе ~30 секунд..."`

Після отримання відповіді — показати фінальне повідомлення (див. розділ 5).

### 3.7 Режим правок

Після побудови агент переходить в стадію `editing`.  
Будь-яке нове повідомлення трактується як правка.

Claude отримує:
```
Таблиця вже побудована. spreadsheet_id: {id}

Поточна структура:
{короткий опис що є}

Користувач хоче:
"{повідомлення користувача}"

Визнач:
1. Що конкретно треба змінити (тип зміни: add_article / remove_article / rename_article / change_access / add_sheet / remove_sheet)
2. Яких даних не вистачає для внесення правки
3. Якщо даних достатньо — сформуй update_payload

Відповідай у форматі JSON:
{
  "changes": [...],
  "missing": [...],
  "update_payload": {...} або null,
  "message_to_user": "..."
}
```

Якщо `missing` не пусте — питає у користувача.  
Якщо `update_payload` є — викликає Apps Script з `action: "update_table"`.

Що побудовано:

## 6. Fallback якщо Apps Script не відповів

1. Повідомити: `"Не вдалось побудувати автоматично. Спробуємо ще раз?"`
2. При повторній невдачі — запропонувати інструкцію вручну
3. Зберегти payload в стані — щоб при наступній спробі не питати заново
4. Команда `/retry` — повторити відправку збереженого payload

---

## 7. Команди

| Команда | Дія |
|---------|-----|
| `/start` | Привітання + пояснення що вміє агент |
| `/clear` | Скинути стан, почати спочатку |
| `/retry` | Повторити відправку payload до Apps Script |
| `/status` | Показати поточний стан агента (для дебагу) |

---

## 8. Що не змінюється від поточної реалізації

- Webhook-архітектура (server.js, webhookHandler.js, bot.js)
- OAuth і service account авторизація (auth.js)
- `GOOGLE_REPORTS_SHARE_MODE=anyone_with_link`
- Legacy режим побудови — залишити як fallback якщо агент не розпізнав тип таблиці
