## 1. Вхідні дані — формат ТЗ від бота 2.2

### Що є зараз
Бот приймає або JSON payload, або вільний текст. JSON парситься як є, текст зливається в чернетку.

### Що треба змінити
Основний вхідний формат — **текстовий блок у code-огортці** (не чистий JSON, не вільний текст).  
Це рядок з Telegram який виглядає так:

```
```tz
business_name: Агенція Ткаченко
business_type: services
team_size: 12
report_type: cashflow

inflows:
  - article: Оплата від клієнтів — SMM
    responsible: Марина
    fixation_moment: Після підтвердження оплати
    ops_per_month: 8
    has_sheets_access: true
  - article: Передоплати
    responsible: Марина
    fixation_moment: В день надходження
    ops_per_month: 3
    has_sheets_access: true

outflows:
  - article: Зарплати
    responsible: Наталія (бухгалтер)
    fixation_moment: День виплати
    ops_per_month: 1
    has_sheets_access: true
  - article: Підрядники на проєктах
    responsible: Дмитро (PM)
    fixation_moment: Після підписання акту
    ops_per_month: 15
    has_sheets_access: false
  - article: Рекламні бюджети клієнтів
    responsible: Ірина (реклама)
    fixation_moment: Щотижня по п'ятницях
    ops_per_month: 20
    has_sheets_access: false

notes:
  - Дмитро і Ірина без доступу до Sheets — потрібна форма
  - Рекламні бюджети — найбільший обсяг операцій
```
```

**Що змінити в парсері:**
- Детектувати code-блок з тегом `tz` (` ```tz `)
- Парсити вміст як YAML-подібний формат (не строгий JSON)
- Якщо тег відсутній але є code-блок — все одно спробувати розпарсити
- Якщо парсинг провалився — запитати уточнення, не падати в дефолтний режим

## 2. Визначення архітектури — нова логіка

### Що є зараз
Бот одразу будує таблиці без етапу визначення архітектури. Є два режими: «за ТЗ» і «legacy фінансові таблиці».

### Що треба додати
**Між отриманням ТЗ і побудовою таблиці — обов'язковий етап діалогу.**

Бот аналізує ТЗ і визначає попередню архітектуру:

| Умова | Архітектура |
|-------|-------------|
| `total_ops <= 50` і всі `has_sheets_access: true` | **A** — прямий ввід в один аркуш |
| `total_ops` від 50 до 200 або є хтось з `has_sheets_access: false` | **B** — окремі вкладки джерел |
| `total_ops > 200` або 3+ людини з `has_sheets_access: false` | **C** — Google Form + лог + скрипт |

`total_ops` = сума всіх `ops_per_month` по всіх статтях.

Архітектури не виключають одна одну — для надходжень може бути A, для витрат C.

**Після визначення архітектури бот повідомляє:**
```
Проаналізував ТЗ. Ось що я бачу:

Загальна кількість операцій: 48/міс
Без доступу до Sheets: Дмитро (15 оп/міс), Ірина (20 оп/міс)

Архітектура:
→ Надходження: режим A (прямий ввід, Марина має доступ)
→ Витрати — Зарплати, Оренда: режим A (Наталія має доступ)
→ Витрати — Підрядники: режим B (Дмитро без доступу → окремий аркуш-форма)
→ Витрати — Реклама: режим C (Ірина без доступу, 20 оп/міс → Google Form)

Перед тим як будувати — кілька питань.
```

І далі задає уточнюючі питання (див. розділ 3).

## 3. Діалог уточнення — без обмеження кількості питань

### Що є зараз
Бот одразу генерує таблиці після отримання ТЗ, без діалогу.

### Що треба додати
Після визначення архітектури бот задає питання — по одному або невеликими групами. Питань стільки скільки потрібно.

**Обов'язкові питання завжди:**
- Google account (email) для створення файлу
- Назва компанії для іменування папки і файлів (пропонує з ТЗ, учасник підтверджує або змінює)
- Яку мову використовувати в таблиці (українська / англійська)

**Питання залежно від архітектури:**

Якщо є `has_sheets_access: false`:
- «[Ім'я] не має доступу до Sheets. Як йому зручніше вносити дані — Google Form чи ти будеш вносити сам за нього?»

Якщо `ops_per_month > 15` по одній статті:
- «[Стаття] — [N] операцій на місяць. [Ім'я] вносить їх одразу після кожної транзакції чи пакетом раз на тиждень?»

Якщо `report_type: cashflow`:
- «Додати аркуш Платіжного календаря? Він покаже прогноз залишку по датах і підготує базу для наступного уроку.»

Завжди:
- «Є кілька банківських рахунків? Якщо так — розводити їх окремо чи зводити в один залишок?»
- «Потрібен облік по контрагентах (хто конкретно платив або кому платили)?»

**Бот не задає всі питання одразу** — групує по 2–3, чекає відповідь, потім якщо є ще — продовжує.

## 4. Структура Google Sheets файлу

### Що є зараз
Legacy режим: Cashflow (аркуші Інструкція, План, Факт, Звіт) і P&L (Інструкція, P&L, Довідник статей).

### Що треба змінити для режиму Cashflow

**Аркуші файлу Cashflow:**

| Аркуш | Наявність | Опис |
|-------|-----------|------|
| 📊 Cashflow | Завжди | Зведений звіт. Формується формулами SUMIF з вкладок-джерел. Захищений від редагування. |
| ⬇️ Надходження | Архітектури B і C | Таблиця: дата / контрагент / стаття / сума / коментар. Стаття — дропдаун з Довідника. |
| ⬆️ Витрати | Архітектура B | Аналогічно Надходженням. |
| ⬆️ Витрати — [Ім'я] | Якщо є особа без доступу (архітектура B) | Окремий спрощений аркуш для конкретної людини. |
| 📅 Платіжний календар | Опціонально | Прогноз залишку по датах. |
| 📝 Лог | Тільки архітектура C | Сирі дані з Google Form. Не редагується вручну. |
| 📋 Довідники | Завжди | Список статей (з ТЗ), відповідальні, контрагенти. Джерело для дропдаунів. |
| ⚙️ Налаштування | Завжди | Назва компанії, місяць, валюта, поріг підсвітки від'ємного залишку. |

**Ключові формули:**
- Зведений Cashflow: `=SUMIF(Надходження!C:C, A2, Надходження!D:D)` по кожній статті
- Дропдауни статей: Data Validation → список з аркушу Довідники
- Умовне форматування: червоний фон якщо залишок < значення з Налаштувань
- Захист: аркуші Cashflow і Довідники — заблоковані від випадкового редагування

## 5. Файлова система на Google Drive

### Що є зараз
Папка `@username - Financial Reports`. Один рівень вкладеності.

### Що треба змінити
Нова структура папок:

```
Google Drive учасника/
└── 📁 Фінансова система — [Назва компанії]/
    ├── 📁 Cashflow/
    │   └── 📊 Cashflow_[Компанія]_[Рік].xlsx
    ├── 📁 P&L/          ← пуста, створюється одразу
    ├── 📁 Баланс/       ← пуста, створюється одразу
    └── 📁 Дашборд/      ← пуста, створюється одразу
```

**Логіка іменування:**
- Папка верхнього рівня: `Фінансова система — [business_name з ТЗ]`
- Файл: `Cashflow_[business_name]_[поточний рік]`
- Бот пропонує ці назви — учасник підтверджує або змінює до створення

**Якщо файл з такою назвою вже існує:**  
Запитати: «Файл вже існує. Перезаписати чи створити новий з датою?»

### Що є зараз
Посилання на папку і файли + статус Validation: OK/FAILED.

### Що треба змінити
Більш детальний і корисний вивід:

📁 Папка: Фінансова система — Агенція Ткаченко
🔗 [Відкрити папку на Drive]

📊 Cashflow_Агенція Ткаченко_2026
🔗 [Відкрити файл]

Що побудовано:
✓ Аркуш «Cashflow» — зведений звіт, формується автоматично
✓ Аркуш «Надходження» — 3 статті, відповідальна Марина
✓ Аркуш «Витрати» — Наталія вносить напряму
✓ Аркуш «Витрати — Дмитро» — спрощений аркуш для PM

Наступні кроки:
1. Відкрий файл і перевір що статті відображаються правильно
2. Надішли Ірині посилання на форму введення
3. Внеси тестову операцію — переконайся що вона з'являється в Cashflow

## 7. Валідація

### Що є зараз
Для legacy режиму — перевірка наявності аркушів і ключових секцій. Для universal режиму — тільки «застосувалось без помилки».

### Що треба додати для Cashflow
- Аркуш «Cashflow» існує і має правильні заголовки
- Аркуш «Довідники» існує і містить список статей з ТЗ
- Формули SUMIF в аркуші «Cashflow» посилаються на існуючі аркуші
- Google Form (якщо є) — посилання валідне і форма відкривається
- Дропдауни в аркушах введення — містять статті з Довідника

## 8. Обробка крайніх випадків

### Що треба додати до поточних обмежень

| Ситуація | Поведінка |
|----------|-----------|
| Учасник не авторизував Drive | Пояснити як надати доступ, повторити спробу |
| Файл з такою назвою вже існує | Запитати: перезаписати або створити новий з датою |
| `has_sheets_access: false` і немає пошти Google | Запропонувати Google Form без авторизації (публічна форма) |
| Більше 50 статей | Попередити що таблиця буде великою, запропонувати об'єднати дрібні |
| ТЗ неповне (відсутні обов'язкові поля) | Не падати в дефолт — задати уточнюючі питання по відсутніх полях |
| API Google повернуло помилку | Поточна поведінка ОК + запропонувати fallback (інструкція вручну) |

## 9. Що залишається без змін

- Webhook-архітектура (server.js, webhookHandler.js, bot.js) — не чіпати
- OAuth і service account авторизація (auth.js) — залишити як є
- Команда `/clear` — залишити
- `/health` і технічні маршрути — залишити
- Legacy режим (фінансові таблиці без ТЗ) — залишити як fallback
- `GOOGLE_REPORTS_SHARE_MODE=anyone_with_link` — залишити
